## REPORT 9 — WORKFLOW RECONSTRUCTION REPORT

### WF-01: Inbound Message Full Routing Workflow

```
INBOUND MESSAGE (WhatsApp / Meta)
│
├─► Bridge Poller (every 1–30s adaptive)
│     ├─ Fetch from SQLite (read-only, UTC-normalized)
│     ├─ LID → Phone resolution
│     ├─ Dedup check (processed_bridge_messages)
│     ├─ OCR if image without text
│     ├─ Voice transcription if audio/ptt
│     ├─ Persist to wbom_whatsapp_messages (ALL DMs, always)
│     └─ Call process_message()
│
├─► Silent-Skip Check
│     ├─ Accountant phone? → SKIP
│     ├─ Name contains skip-token? → SKIP
│     └─ Role = blocked? → SKIP
│
├─► detect_identity(phone, text)
│     └─ Returns role + confidence + source
│
├─► Role Routing (priority order)
│     1. family → personal reply
│     2. escort_roles → handle_escort_client_message()
│     3. admin → command/NL/help
│     4. attendance detected → create_attendance_draft()
│     5. Intent classification (LLM → fallback deterministic)
│     6. accountant → payment ingest / KB / AI
│     7. candidate/unknown → recruitment_eligibility()
│     8. escort order intent → handle_escort_client_message()
│     9. employee → verification/attendance/slip/release/complaint/advance
│     10. advance request (any) → start_advance_verification()
│     11. office_location → KB fast path
│     12. KB lookup
│     13. Reviewed reply memory
│     14. AI fallback (GitHub Models → Groq → Ollama)
│
└─► Reply Delivery
      ├─ Check auto_reply_enabled
      ├─ Check draft-always gate
      ├─ Check safe intents
      ├─ Check complaint phrases
      ├─ Check cooldown (60s Redis/in-memory)
      ├─ Check loop detection
      ├─ Check keyword flood
      ├─ Check prompt injection
      ├─ Check outbound poison
      └─ SEND or DRAFT
```

### WF-07: Payroll Workflow

```
SCHEDULER (02:00 Asia/Dhaka daily)
│
├─► compute_all_for_period(year, month)
│     └─ For each Active employee:
│           compute_run(employee_id, year, month)
│           ├─ Idempotency: skip if existing active run exists
│           ├─ Count completed escort programs in month
│           ├─ Calculate: basic + (days × ৳800) - advances
│           ├─ Write wbom_payroll_runs (status='draft')

│     draft → reviewed (submit_run)
│     reviewed → approved (approve_run)

│     locked → paid (mark_paid with method + reference)
│     any → cancelled (cancel_run + reason)
│
└─► Each transition writes wbom_payroll_approval_log
```

## REPORT 10 — STATE MACHINE REPORT

### SM-01: Payroll Run State Machine

```
                    cancelled
                   ↗ ↗ ↗ ↗
draft → reviewed → approved → locked → paid
```

| Transition | Function | Required Actor |
|---|---|---|
| draft → reviewed | `submit_run()` | operator+ |
| reviewed → approved | `approve_run()` | admin+ |
| approved → locked | `lock_run()` | admin+ |
| locked → paid | `mark_paid()` | accountant+ |
| any → cancelled | `cancel_run()` | accountant+ |

### SM-02: Escort Program State Machine

```
draft → confirmed → Assigned → Running → Completed
                 ↘ Cancelled (from any pre-Completed)
```

| State | Meaning |
|---|---|
| draft | Client order received, escort not yet assigned |
| confirmed | Admin assigned escort via ESCORTCONFIRM |
| Assigned | Program assigned, escort not yet on site |
| Running | Escort on active duty |
| Completed | Release confirmed, attendance backfilled, payment drafted |
| Cancelled | Program cancelled |

### SM-03: Payment Draft State Machine

```
pending → sent (after admin PAID/ADVANCE command)
pending → rejected (after admin REJECT command)
pending → expired (after 24h TTL cleanup job)
```

### SM-04: Recruitment Session State Machine

```
new → collecting (step 1-6) → scored
                             ↘ expired (24h TTL)
```

### SM-05: Attendance Draft State Machine

```
pending → approved → saved to wbom_attendance
        → rejected
```

### SM-06: Escort Roster Entry State Machine

```
(created) → completed (on lifecycle close)
```

### SM-07: Draft Reply State Machine

```
pending → approved → sent_at timestamp set
        → rejected → status='rejected'
        → expired (draft_ttl_cleanup job, default TTL from .env)
```

### SM-08: Admin (User) State Machine

```
active → disabled (user_remove command)
```

## REPORT 21 — RAG REPORT

### Architecture
- **Algorithm:** BM25 (offline, no external embedding service)
- **Language:** Bilingual: Bangla (`\u0980-\u09FF`) + English
- **Index:** In-memory, rebuilt on demand
- **Rebuild:** Daily at 18:00 (scheduler) + on-demand API

### Index Sources
1. Plain-text files under `core/resources/*.txt` (excluding internal dirs and files)
2. Active rows from `fazle_knowledge_base` (key + reply_text)

### BM25 Parameters
- k1 = 1.5, b = 0.75
- Chunk size: 320 characters, overlap: 60 characters
- Min token length: 2 characters

### Safety Filters (3 layers)
1. **Directory exclusion:** `_internal_archived`, `_internal`, `prompts`, `debug`, `tests`, `drafts`, `internal`, `ai`, `training`, `examples`, `temp`
2. **Filename keyword exclusion:** analysis, prompt, intent, debug, test, sample, chain, reasoning, internal, archived, system_context
3. **Chunk-level safety:** 30+ internal marker patterns — unsafe chunks **purged entirely** from index (never stored, `unsafe_docs` always = 0)

### Stop Words
80+ Bangla and English function words removed before indexing. Critical note: `লোক`, `পদে`, `নেওয়া`, `হচ্ছে` were originally stop words but **removed** because they are content words in recruitment queries like `কোন পদে লোক নেওয়া হচ্ছে`.

### Search Behaviour
- Query tokenized same way as documents
- Scores via BM25 per document
- Unsafe chunks skipped during scoring
- Top-k returned with source traceability
- Last 50 queries in ring buffer for debugging (`recent_searches()`)
