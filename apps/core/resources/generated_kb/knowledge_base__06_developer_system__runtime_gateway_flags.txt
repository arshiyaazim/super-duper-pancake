## IMPORTANT — Correction to `kb_enrichment_plan_v2.md` P1-F

`kb_enrichment_plan_v2.md` P1-F states: *"All feature flags in `shared/runtime_gateway.py`."*

This is **incorrect.** `shared/runtime_gateway.py` is a **distributed node registry and heartbeat manager**, not a feature-flag registry. It manages `fazle_runtime_nodes` (DB table) and the heartbeat lifecycle of three registered app instances. It contains no feature flags.

Feature flags live in **`.env`** (and are loaded into `app/config.py`). They are read by individual modules. This article documents both: (1) all production feature flags from `.env`, and (2) the `shared/runtime_gateway.py` node system.

## Section 2 — Runtime Node System (`shared/runtime_gateway.py`)

This module manages the **distributed node registry**. Three separate app processes register as nodes:
- `fazle-core` — main WhatsApp bot and API
- `payroll-engine` — payroll computation service
- `escort-roster` — escort program sync service

---

### `RuntimeNode` Dataclass

Source: `shared/runtime_gateway.py` lines 109–121

```python
@dataclass
class RuntimeNode:
    node_id:         str                 # "{app_name}-{hostname}-{pid}"
    app_name:        str                 # "fazle-core" | "payroll-engine" | "escort-roster"
    role:            str                 # "orchestrator" | "worker" | "standalone"
    status:          str   = "online"    # "online" | "offline" | "degraded"
    last_seen:       float = time.time() # Unix timestamp of last successful heartbeat
    version:         str   = "unknown"   # App version string
    active_requests: int   = 0           # In-flight requests at heartbeat time
    queue_depth:     int   = 0           # Queue depth at heartbeat time
    metadata:        dict  = {}          # Arbitrary JSON for diagnostics
```

**Node ID format:** `{app_name}-{hostname}-{pid}` (example: `fazle-core-vmi3117764-12345`)

The PID component makes each node ID unique across restarts on the same host. A crashed-and-restarted process generates a new node_id.

---

### Heartbeat & Stale Detection

| Parameter | Value | Effect |
|---|---|---|
| `HEARTBEAT_INTERVAL_S` | **30 seconds** | How often each node updates `fazle_runtime_nodes.last_seen` |
| `STALE_THRESHOLD_S` | **90 seconds** | `last_seen > 90 s ago` → node is marked `offline` by stale sweep |
| `_STALE_SWEEP_EVERY` | **3 cycles** | Stale sweep runs every 3 heartbeats = every 90 seconds |

**Stale sweep is atomic:** Uses `UPDATE ... RETURNING` on `fazle_runtime_nodes` — no lost-update race between multiple nodes running the sweep concurrently.

---

### Reconnect Alerting

`heartbeat_loop()` tracks consecutive failures. At **3 consecutive failures**, it emits a `RECONNECTING` event. On the next successful heartbeat, it emits `WS_RECOVERED`.

---

### What Happens When a Node Dies

1. The dying node calls `stop_gateway()` in its lifespan teardown — marks itself `offline` immediately
2. If the crash is unclean (no teardown): stale sweep detects `last_seen > 90 s` → marks offline
3. `mark_stale_nodes()` also calls `_release_claimed_queue_items()` — releases any `fazle_message_queue` items with `processor_id = dead_node_id` back to `pending` status (attempts capped at 2 to avoid immediate dead-letter)

---

### Core Functions

| Function | Purpose |
|---|---|
| `start_gateway(app_name, role, version, ...)` | Register node + start heartbeat task (call from app lifespan startup) |
| `stop_gateway()` | Cancel heartbeat task + deregister node (call from app lifespan teardown) |
| `register_node(app_name, role, version, ...)` | UPSERT into `fazle_runtime_nodes`; returns node_id |
| `heartbeat(node_id, active_requests, queue_depth)` | Touch `last_seen`; returns True/False |
| `deregister_node(node_id)` | Mark node `offline` (clean shutdown) |
| `mark_stale_nodes(stale_after_seconds=90)` | Atomic stale sweep; releases queue items |
| `get_active_nodes()` | Return all nodes with `age_s` field; used by `/api/runtime/nodes` |
| `get_current_node_id()` | Return this process's node_id (None if start_gateway not called) |

### Runtime Event Bus

`shared/runtime_gateway.py` re-exports `shared.events` as `shared_runtime_event_bus`. Events:

| Event Constant | Trigger |
|---|---|
| `NODE_ONLINE` | `register_node()` success |
| `NODE_OFFLINE` | `deregister_node()` (clean) or `mark_stale_nodes()` (stale) |
| `QUEUE_PRESSURE` | `_release_claimed_queue_items()` releases > 0 items |
| `STALE_FRONTEND` | (defined in shared.events; emitted by other callers) |
| `LOCK_CONTENTION` | (defined in shared.events; emitted by other callers) |
| `RECONNECTING` | 3 consecutive heartbeat failures |
| `WS_RECOVERED` | Heartbeat success after failures |

## Section 3 — Safe Toggle Procedures

### Flags Safe to Toggle Without Restart

These flags are read on each message processing cycle:

| Flag | Toggle Method | Notes |
|---|---|---|
| `AUTO_REPLY_ENABLED` | Edit `.env` → restart process (or use app config reload if implemented) | Most common toggle — switches between draft-only and live-send mode |
| `AI_SAFE_MODE` | Edit `.env` → restart process | Use in degraded-trust situations |
| `DRAFT_QUALITY_GATE` | Edit `.env` → restart process | Disable only for debugging |

### Emergency Disable Sequence

**Scenario: Auto-reply storm (bot sending wrong messages)**
1. Set `AUTO_REPLY_ENABLED=false` in `.env`
2. Restart `fazle-core`: `pm2 restart fazle-core` (or equivalent)
3. All subsequent replies will queue to `fazle_draft_replies` instead of sending
4. Inspect queue: `DRAFTS` WhatsApp admin command or `/api/drafts` endpoint
5. After fixing root cause: set `AUTO_REPLY_ENABLED=true` → restart

**Scenario: Outbound queue overflow / send failure**
1. Set `OUTBOUND_ENABLED=false` → restart
2. Queue accumulates in `fazle_message_queue` (no messages lost if queue persists)
3. After fixing outbound service: set `OUTBOUND_ENABLED=true` → restart

**Scenario: LLM provider down (Ollama / Groq / GitHub Models)**
1. Set `PRIMARY_AI_PROVIDER=ollama` in `.env` → restart
2. Ollama runs locally at `OLLAMA_URL` and is the primary provider
3. If Ollama is degraded: fallback proceeds to Groq, then GitHub Models
4. If Ollama must be bypassed for replies: set `OLLAMA_REPLY_DISABLED=true` → restart
4. System falls back to KB-only replies (no LLM output)

**Scenario: Hybrid RAG causing wrong results**
1. Set `HYBRID_SEARCH_ENABLED=false` → restart
2. RAG falls back to BM25-only mode
3. Vector search (Qdrant) is bypassed; BM25 index still available

## Key Relationships

```
.env (all feature flags)
  │
  ├── AUTO_REPLY_ENABLED ──→ bridge_poller: skip outbound if false
  ├── HYBRID_SEARCH_ENABLED ──→ modules/rag: use BM25 only if false
  ├── DRAFT_QUALITY_GATE ──→ modules/draft_quality: bypass all checks if false
  ├── OUTBOUND_ENABLED ──→ modules/outbound: drop sends if false
  ├── SCHEDULER_ENABLED ──→ modules/scheduler: no jobs run if false
  ├── REVIEWED_REPLY_MEMORY_ENABLED ──→ message_router step 14: skip lookup if false
  ├── DRAFT_ALWAYS_PHONES / ROLES / NAMES / PREFIXES ──→ bridge_poller: force draft
  └── AI_SAFE_MODE ──→ bridge_poller: force draft on long/low-conf/unclear replies

shared/runtime_gateway.py
  │
  ├── fazle_runtime_nodes (DB table) ── node health registry
  ├── fazle_message_queue (DB table) ── queue items released on node death
  └── shared.events ── event bus for NODE_ONLINE/OFFLINE/QUEUE_PRESSURE
```
