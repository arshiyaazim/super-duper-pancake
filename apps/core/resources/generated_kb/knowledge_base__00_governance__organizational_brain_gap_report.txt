## Critical Conflicts (Must Resolve Before Wave-3)

### CONFLICT-1 — Draft TTL Mismatch
- **KB says:** `automation_pipeline.md` — "Expire old pending drafts after 48 hours unless admin marks hold"
- **Production says:** `.env` — `DRAFT_TTL_HOURS=24` (24 hours)
- **Impact:** Admin staff expect 48h window to approve drafts. Production expires them in 24h. Support staff will report "missing drafts" as bugs.
- **Action:** Management to confirm authoritative value → update KB to match production, or update production to match KB policy.
- **Priority: CRITICAL**

### CONFLICT-2 — Hybrid RAG KB Documents Wrong Algorithm
- **KB says:** `hybrid_search.md` (17 lines) — describes 5-signal ranking: "Role match > keyword > semantic > recency > workflow state"
- **Production says:** RRF fusion: `score = 1/(60+rank_bm25) + 1/(60+rank_vector)` — pure rank fusion, no role or recency signals
- **Impact:** Any engineer reading the KB to understand the RAG system will implement the wrong algorithm. Incident response will be misdirected.
- **Action:** Rewrite `hybrid_search.md` with actual Qdrant + MiniLM + RRF architecture (see `kb_enrichment_plan_v2.md` PRIORITY-1).
- **Priority: CRITICAL**

### CONFLICT-3 — Age Rule Divergence Across Channels
- **KB (BR-25) says:** Management decision — age range 18–55 for all recruitment
- **Social auto-reply says:** No age restriction applied in Facebook/Messenger auto-replies
- **Impact:** A candidate told "18-55 only" via WhatsApp may receive a conflicting auto-reply via Facebook with no age requirement. Dispute risk.
- **Action:** Management to decide whether Facebook auto-reply should enforce BR-25 age restriction. Update `social_auto_reply_system.md` to document the decision, or update `social_auto_reply/reply_rules.py` to match.
- **Priority: CRITICAL**

### CONFLICT-4 — Dual Escort Draft TTL Mechanisms (Clarification of CONFLICT-1)
- **escort_roster/db.py says:** `expire_stale_drafts(hours=48)` — hardcoded 48-hour expiry for **escort roster draft entries** (type: `escort_roster_entries` rows with status='draft')
- **Production .env says:** `DRAFT_TTL_HOURS=24` — 24-hour expiry for **AI reply drafts** (type: `fazle_draft_replies` rows — drafts awaiting admin approval for WhatsApp send)
- **These are DIFFERENT tables governing DIFFERENT draft types.** The 48h and 24h values do NOT apply to the same objects. However:
  - The KB does not distinguish between these two draft systems
  - `automation_pipeline.md` says "expire after 48 hours" without qualifying which draft table
  - Operators who read the KB may apply the 48h expectation to WhatsApp reply drafts (which expire in 24h)
- **Impact:** Operator confusion and incorrect escalation paths. Low severity but creates audit ambiguity.
- **Action:** Update `automation_pipeline.md` to clearly distinguish the two draft systems: escort roster drafts (48h TTL, `escort_roster_entries`) vs reply drafts (24h TTL per DRAFT_TTL_HOURS env var, `fazle_draft_replies`).
- **Priority: MEDIUM (clarification, not a logic error)**

### CONFLICT-5 — Escort Duty Rate Discrepancy: ৳800 vs ৳1,200/day
- **`modules/payroll/__init__.py` says:** `DEFAULT_PER_PROGRAM_RATE = 800.0` — default escort program daily pay rate used in monthly payroll run computation
- **`modules/payment_workflow/__init__.py` says:** `DEFAULT_DAILY_RATE = 1200.0` — default daily rate used in payment workflow (individual advance/payment requests)
- **Two different live modules use two different hardcoded daily rates.** Both are in active code paths.
  - `payroll` module: used when computing `wbom_payroll_runs` (monthly batch)
  - `payment_workflow` module: used when an employee requests payment for a specific program
- **Impact:** An employee may receive ৳800/day in monthly payroll but their individual payment request calculates at ৳1,200/day, or vice versa. **Financial discrepancy risk. Direct employee payment disputes.**
- **Action:** IMMEDIATE management decision required. Authoritative escort duty rate must be set, documented in KB, and both constants updated to match. This is a **financial integrity issue**.
- **Priority: CRITICAL — Management Authorization Required**
