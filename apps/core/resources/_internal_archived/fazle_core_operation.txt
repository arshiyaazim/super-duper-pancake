# JARVIS — Autonomous AI Command Platform
## Vision
A single-VPS, Docker-native, local-first AI platform that acts as a personal Jarvis: answering phone calls, managing security business operations, handling employee/client relationships, generating invoices, and reasoning autonomously — all through one unified interface.
---
## Architecture Philosophy: "Lean & Smart"
**5 Core Principles:**
1. **One Brain, Many Agents** — Single Ollama model (`qwen2.5:1.5b`) + GPT-4o fallback, orchestrated by 9 specialized agents
2. **Minimal Containers, Maximum Capability** — ~12 Docker services (not 40), each doing more
3. **DB-First Intelligence** — PostgreSQL + pgvector replaces Qdrant; one DB does semantic memory + relational + vector
4. **Event-Driven, Not Polling** — WebSocket + SSE for real-time; no busy loops
5. **Mobile-Native Voice** — LiveKit WebRTC + Twilio SIP for sub-second Jarvis-style voice

## Voice Flow (Jarvis Call Handling)
```
Inbound Call → Twilio SIP → LiveKit Room → VoiceAgent
    → STT (Whisper/Deepgram) → ConversationAgent + MemoryAgent
    → LLM (Ollama/GPT-4o) → ToolAgent (if action needed)
    → TTS (ElevenLabs/Kokoro) → LiveKit → Twilio → Caller
```

1. PROJECT OVERVIEW
I want to build a web application to manage the following:
1.	Employee Records – Maintain personal and salary-related data for each escort employee.
2.	Mother Vessel–wise Records – Track escort assignments per Mother Vessel for bill generation.
3.	Employee Name–wise Records – Track each employee's assignments for salary calculation.

2. DATA SOURCES & INPUT METHODS
2.1 Order Reception (via WhatsApp Messages)
I receive escort service orders through WhatsApp messages. These messages vary in format but contain consistent data points.
________________________________________
2.2 Message Parsing Rules
Key Data Points to Extract
Field	Description
Mother Vessel	The main cargo ship (e.g., MV F Glory, MV Atlantic Ocean)
Lighter Vessel	The smaller transport vessel (e.g., MV Meta-1, Malekul Bahar-2)
Master Mobile Number	The lighter vessel master's phone number
Destination Point	Where the lighter vessel is headed (e.g., Rupshi, N. Para, Noapara)
Fields to Ignore
•	Importer's name (e.g., Nabil)
•	Product name (e.g., Wheat, Soybeans Meals In Bulk)
•	Lighter vessel capacity (e.g., 1200 M.T., 2500 Mts)
Structural Parsing Rule
Lighter Vessel Name, Mobile Number, and Destination Point — these three pieces of information always stay close together.
Mother Vessel Name stays separate from that group — either at the beginning or the end of the message.
Additional Parsing Rules
•	If there is one mobile number, it means one lighter vessel needs escort service.
•	If there are multiple mobile numbers, each corresponds to a separate lighter vessel assignment.
•	The text "# [Lighter Name] cancel" means a previously assigned lighter vessel is cancelled and replaced by a new lighter vessel in the same message.
•	A leading '0' in any number must always be preserved (e.g., 01782610995 must not become 1782610995).

5. DATA RECORDING — PROGRAM DATA SHEET
5.1 Overview
After generating the reply message, the extracted and filled data must be saved to a Program Data Sheet (similar to an Excel file I have provided).
5.2 Required Columns
Column	Description	Source
ID	Auto-generated unique ID (if not pre-existing). If ID already exists, a new row is still created.	System
Mother Vessel	Name of the mother vessel	Parsed from order message
Lighter Vessel	Name of the lighter vessel	Parsed from order message
Master Mobile	Lighter vessel master's phone number	Parsed from order message
Destination / Release Point	Where the lighter is going	Parsed from order message
Escort Name	Name of assigned escort employee	Manually entered
Escort Mobile	Mobile number of the escort	Manually entered
Date	Assignment date	Auto from reply generation
Day/Night	(D) or (N)	Auto from time of reply
(Additional columns)	As per the attached Excel file structure	From Excel
5.3 Data Retrieval / Search
Records can be fetched/searched by:
•	ID
•	Mother Vessel
•	Employee Name
•	Lighter Vessel
•	Release Point
•	Date
5.4 Incomplete Records
The reply message only provides half the data. The record is completed only when the employee is released from the destination point and their food bill and conveyance are processed.

6. DUTY COMPLETION & PAYMENT — SEND MONEY SYSTEM
6.1 Overview
When an escort employee completes their duty and is released from the destination point, I need to:
1.	Send money for food bill and conveyance.
2.	Update the Program Data Sheet (if duty is completed).
3.	Update the Daily Cash Payment Sheet.
6.2 Send Money Message Template
text
ID: [Employee Mobile/ID]
[Employee Name]
[Receiving Mobile Number]([Payment Method])
[Amount]/-,
Status: Running/Completed,
Remarks: [Optional text]
Where:
Field	Description
Employee ID	The employee's mobile number serves as their ID
Employee Name	Full name of the escort
Receiving Mobile	The number money is sent to
Payment Method	(N) = Nagad, (B) = bKash (indicated in parentheses)
Amount	Amount in BDT
Status	Running = duty ongoing; Completed = duty finished
Remarks	Optional notes
6.3 Update Logic
Status in Message	Action
Running	Update Daily Cash Payment Sheet only
Completed	Update both Program Data Sheet and Daily Cash Payment Sheet (with all column-wise information and applicable calculations)

10. TECHNICAL REQUIREMENTS SUMMARY
10.1 Core Features
#	Feature	Description
1	Chat Interface	Paste WhatsApp messages to parse order/payment data
2	Message Parser	AI/rule-based engine to extract structured data from unstructured messages
3	Reply Template Generator	Auto-generate reply messages with date, time (D/N), and company name
4	Program Data Sheet	Excel-like data table for all escort assignments
5	Daily Cash Payment Sheet	Track all payments sent to employees
6	Employee Records	Master list of all escort employees with salary tracking
7	Mother Vessel Records	Group all assignments by Mother Vessel for billing
8	Search & Filter	By ID, Mother Vessel, Employee, Lighter Vessel, Release Point, Date
9	Excel Export/Import	Read from and write to Excel format; preserve leading zeros
10	ID Auto-Generation	Auto-create unique ID if none exists; allow existing IDs to create new rows
10.2 Data Integrity Rules
•	Leading zeros in phone numbers and IDs must always be preserved.
•	Phone numbers with +88 prefix should be normalized to 0-prefix format.
•	Payment method codes: (N) = Nagad, (B) = bKash (case-insensitive).
•	Date format: DD.MM.YYYY in reply messages.
•	Amounts: Parse regardless of formatting (e.g., 1000/-, = 200/-, 1500/-).
10.3 Sheets / Tables Required
1.	Program Data Sheet — Full escort assignment records (from attached Excel structure + ID column)
2.	Daily Cash Payment Sheet — All money sent, with date, employee, amount, method, status
3.	Employee Master Sheet — Employee name, mobile (ID), salary info
